# hmall 项目简历埋点

## 一、先给结论：这个项目够不够拿去面试？

**结论：够用，甚至对实习生来说已经偏深。**

但前提是：

1. **你不能把它吹成“金融级绝对一致”系统**，因为从实际代码看，你的下单链路本质上仍然是**最终一致性**。
2. **你要主动承认中间态存在**，但同时说明你做了哪些工程手段把中间态的窗口、影响面、资损风险压到很小。
3. **你要把“我知道这个方案不是银弹”讲出来**，面试官反而会觉得你清醒。

如果你像之前那样直接说“彻底解决”“绝对一致”“不可接受场景也完全没问题”，很容易被面试官抓住“那你下单成功但订单还没落库这几百毫秒算什么？”一路往死里追问。

所以，这个项目最好的讲法不是“我做了一个完美系统”，而是：

> 我在一个电商交易场景里，围绕高并发下单、库存一致性、消息可靠投递、缓存治理和压测定位，做了一套**最终一致性优先、吞吐优先、工程上可落地**的方案；我也清楚它与强一致方案的边界。

## 二、基于代码核实后的真实技术画像

以下内容是**根据你当前代码能站得住脚的表述**，不是空泛八股。

### 1. 交易主链路：Redis 预扣库存 + 异步落库 + Outbox 补偿

从 `trade-service/src/main/java/com/hmall/trade/service/impl/OrderServiceImpl.java` 可以确认：

- 下单入口 `createOrder()` 先生成 `orderId`
- 通过 Lua 在 Redis **原子执行**：
  - 扣减库存
  - 写入 Redis outbox 暂存消息
- 然后尝试实时投递 RabbitMQ
- 如果 RabbitMQ 发送失败，消息仍然留在 Redis / MySQL outbox 中，后续由 XXL-Job 补偿

这说明你的核心设计思路是：

> **把同步强耦合链路拆成“前台快速返回 + 后台异步收敛”的最终一致性架构。**

这点是成立的。

### 2. 订单落库采用本地事务 + MySQL 本地消息表

`handleDbOrder()` 里能确认：

- 订单主表、订单明细、本地消息表 `local_event_outbox` 在**同一个本地事务**里提交
- `afterCommit()` 中尝试实时发 MQ
- 失败则依赖 `ItemDeductJobHandler` 等 XXL-Job 继续扫描 `local_event_outbox` 补发

这就是比较典型的：

> **Outbox Pattern + Best Effort 实时投递 + 定时补偿兜底**

这部分是你项目里最像“大厂工程味”的一段，可以重点讲。

### 3. 消费端幂等不是靠嘴说，而是靠“唯一键/状态机/幂等拦截”兜底

从代码看，幂等手段包括：

- `OrderMQListener` 中消费异步订单时，直接以 `orderId` 为主键落库
- `handleDbOrder()` 里先 `getById(orderId)` 做一次前置幂等判断
- 即使绕过前置判断，数据库主键冲突还会触发 `DuplicateKeyException`，监听器捕获后直接 ACK
- `markOrderPaySuccess()` / `cancelOrder()` 都采用了**带前置状态约束的 update**，防止状态乱跳
- Redis 库存恢复用了 Lua + `setnx` 防重放凭证
- `stock_deduct_with_version.lua` / `stock_restore_with_version.lua` 中还加入了版本校验与操作幂等 Key

因此你可以稳妥地说：

> 我的幂等不是只做一层，而是做了 Redis / DB / 状态机多层防重，避免 MQ 重复投递、超时重试、消费者重启导致重复执行。

### 4. 库存一致性是“Redis 主承压，MySQL 兜底对账”思路

你代码里真实存在的能力：

- Redis Lua 原子扣减/恢复库存
- 库存版本号控制：`stock_deduct_with_version.lua`、`stock_restore_with_version.lua`
- `StockReconciliationJobHandler` 做 MySQL 与 Redis 的离线对账修复

因此可以讲成：

> 高频库存路径优先走 Redis 原子脚本抗并发，MySQL 作为最终事实源，通过定时对账修复极端情况下的偏差。

但注意：

### 5. 你简历里“Canal CDC、BloomFilter、ZSet+MGET 分层缓存”目前**代码证据不够强**

我根据代码检索后，当前仓库里**没有明显搜到**：

- Canal / Binlog CDC 的直接实现代码
- BloomFilter 的直接接入代码
- 你描述中的“首页 ZSet 索引 + MGET 详情分页缓存”的核心实现代码

所以如果你要写进简历，必须非常谨慎：

- 如果这些是你本地另一个分支/未提交代码里确实做过，可以写

否则一旦面试官问：

## 五、你必须准备好的核心埋点

下面这些点，是面试官最容易顺着问下去的。

---

### 埋点 1：为什么不用 Seata AT，而改成最终一致性？

#### 推荐回答

因为我的场景是高并发下单，核心诉求是：

1. **入口吞吐不能被同步 RPC 拖死**
2. **不能把多个服务绑进一个长事务里**
3. **要能扛住单点服务抖动、MQ 重投和进程重启**

Seata AT 的问题在于：

- 全局事务链路更长
- MySQL Undo Log / 全局锁开销更大
- 高并发下热点数据更容易被锁住
- 一旦某个下游慢，主链路 RT 会被显著放大

所以我选了：

> 本地事务保证单服务内一致性，跨服务用 MQ + 幂等 + 补偿实现最终一致性。

#### 追问：那你这个方案缺点是什么？

**必须主动承认：**

- 存在中间态，例如“Redis 已预扣，但订单还没最终落库”
- 需要补偿任务兜底，系统复杂度更高
- 对幂等、重试、状态机设计要求更高
- 不适合“绝对不能看到中间态”的金融强一致场景

这句话特别重要：

> 如果是支付记账、账户余额、证券撮合这类强一致场景，我不会用这套方案直接上；但对电商下单场景，行业里更常见的是以最终一致性换吞吐和可用性。

这句很加分。

---

### 埋点 2：你这个下单是不是会有“用户看到成功，但订单还不存在”的中间态？

#### 标准回答

会有短暂中间态，这个是最终一致性方案的客观事实。

但我做了三层收敛：

1. **前台返回成功的前提**是 Redis 预扣库存成功，先保证库存不会被超卖
2. **后台异步落库**，订单主表 + 明细 + Outbox 在一个本地事务中提交
3. **实时投递失败时由 XXL-Job 扫描 Outbox 补偿**，保证消息不会因为服务抖动直接丢失

所以我的目标不是“完全没有中间态”，而是：

> 把中间态控制在一个可恢复、可追踪、可补偿的窗口内。

#### 如果面试官继续压

你可以补一句：

> 我不会把这个方案包装成强一致；它更适合电商交易里“先锁资源、再异步收敛订单状态”的场景。如果业务方明确要求用户在任何时刻都不能读到中间态，那要么引入更强的事务模型，要么前台就不能在预扣成功后立即返回。

---

### 埋点 3：Outbox Pattern 在你这里具体怎么落地？

#### 标准回答

我这里分成两层：

1. **Redis outbox**：下单入口的 Lua 原子执行里，把“库存预扣”和“待发送消息暂存”放在一起，防止 Redis 和 MQ 双写不一致。
2. **MySQL outbox**：订单异步落库时，把订单主表、明细和 `local_event_outbox` 放在同一个本地事务里提交；`afterCommit` 尝试实时投递，失败后由 XXL-Job 扫表补发。

目标分别是：

- Redis outbox 解决入口阶段“预扣成功但消息没发出去”
- MySQL outbox 解决订单已落库但“通知库存/购物车/延迟关单消息没发出去”

#### 追问：为什么要两层 outbox？

因为你链路分成了两个阶段：

- **阶段 1：入口接单 / 预扣库存**
- **阶段 2：订单正式落库 / 下游事件派发**

这两个阶段的事实源不同：

- 阶段 1 更依赖 Redis 的高并发原子能力
- 阶段 2 更依赖 MySQL 事务提交结果

所以各自需要适合自己的可靠消息机制。

### 埋点 4：你说消费端绝对幂等，真能做到“绝对”吗？

不建议你说“绝对”。

建议改口为：

> 我做的是**工程上的强幂等兜底**，尽量把重复执行收敛成无害操作。

#### 你实际可讲的幂等点

- 订单消费：`orderId` 主键唯一

所以简历最好不要写得太死。

建议写成：

### 埋点 5：订单超时关闭为什么还要查支付中心？

这是个很容易加分的点。

#### 推荐回答

因为延迟消息到达时，本地订单状态还是未支付，并不等于用户真的没付钱。

存在一种竞态：

- 用户刚完成支付
- 支付回调由于网络抖动尚未同步到订单服务
- 这时延迟关单消息先到了

如果我只看本地订单状态，就可能把**实际已支付订单误关掉**。

所以我在 `OrderDelayMessageListener` 里先查支付中心：

- 如果支付中心已支付，就修正订单状态为已支付
- 如果支付中心也未支付，再执行关单和逆向补偿

这本质上是一个**防止误关单的最终校验**。

### 埋点 7：GUI 压测为什么把 cart-service 打崩了？

你给的 `hs_err_pid41080.log` 已经很有说服力。

#### 从日志能明确读出的关键信息

- 崩溃服务：`com.hmall.cart.CartApplication`
- 错误类型：**JVM Native Memory Allocation Failure**
- 具体报错：
  - `There is insufficient memory for the Java Runtime Environment to continue`
  - `Native memory allocation (mmap) failed to map 645922816 bytes for G1 virtual space`
- JVM 正在执行：`G1CollectForAllocation`
- Java 线程数：约 **384**

#### 这说明什么？

这不是简单一句“Java heap OOM”。

更准确地说：

> JVM 在向操作系统申请本地内存时失败了，表现为 native memory 不够，可能由堆内存、线程栈、本地缓冲区、JIT/code cache 等共同挤压导致。

对面试官建议这么说：

- GUI 模式下，JMeter 本身就比命令行模式更吃资源
- 本机同时跑压测工具、多个微服务、IDE，会进一步挤占内存
- cart-service 当时线程数已经很多，线程栈和 JVM native memory 压力上升
- 最终在 G1 为分配/扩容申请虚拟空间时失败，服务直接崩溃

#### 能再往深说一点吗？

可以：

> 这个问题不一定是“代码里 new 对象太多”那么简单，也可能是**线程数过多导致栈内存膨胀**，或者在 Windows 本机压测时，JMeter GUI、IDEA、多个 JVM 竞争物理内存，最终让某个服务在 native memory 侧先失败。

这比只说“内存溢出了”专业很多。

### 埋点 8：如果 MySQL 被打崩，会出现什么报错？怎么判断？

这个是典型大厂面试追问。

你不能只说“连接不上”。

#### 1. 应用层可能看到的典型报错

在 Spring Boot + MyBatis / Feign 场景里，常见有：

- `Communications link failure`
- `The last packet successfully received from the server was xxx milliseconds ago`
- `CannotGetJdbcConnectionException`
- `HikariPool - Connection is not available, request timed out`
- `MySQLNonTransientConnectionException`
- `Lock wait timeout exceeded`
- `Deadlock found when trying to get lock`
- `Too many connections`
- `Read timed out`
- `Connection reset`

#### 2. 不同故障含义要区分

##### 场景 A：MySQL 直接宕机 / 网络断开

表现：

- 应用拿不到连接
- 连接池超时
- 报 `Communications link failure` / `CannotGetJdbcConnectionException`

##### 场景 B：MySQL 没宕，但被打满

表现：

- SQL RT 明显抖高
- Hikari 连接池耗尽
- 报 `Connection is not available, request timed out`
- 慢 SQL 增多

##### 场景 C：热点更新严重锁冲突

表现：

- `Lock wait timeout exceeded`
- `Deadlock found when trying to get lock`

##### 场景 D：连接数被打满

表现：

- `Too many connections`

#### 3. 你怎么判断到底是哪一种？

标准回答是“三层排查”：

##### 第一层：看应用日志

看异常类型：

- 是获取连接失败？
- 是 SQL 执行超时？
- 是锁等待超时？
- 是死锁？

##### 第二层：看连接池与线程池表现

例如：

- Hikari 活跃连接数是否打满
- 获取连接等待时间是否持续升高
- 应用线程是否大量阻塞在 DB I/O 上

##### 第三层：看 MySQL 侧指标

例如：

- `Threads_connected`
- `Threads_running`
- `max_connections`
- 慢查询日志
- `show processlist`
- InnoDB 行锁等待
- CPU / IO util

#### 4. 面试里怎么结合你的项目说？

你可以说：

> 如果我的订单异步落库阶段 MySQL 被打崩，最直接的现象通常不是前台 HTTP 立刻全部报错，而是 MQ 消费端的落库逻辑开始异常，监听器捕获异常后执行 NACK，消息不断重回队列；应用日志里会出现获取连接失败、连接池超时或 SQL 超时，队列堆积上升，订单中间态停留时间也会变长。这时就能判断问题在 DB 落库阶段，而不是前面的 Redis 预扣阶段。

这段很有面试价值。

### 埋点 9：你的系统怎么判断是 Redis 顶不住、MQ 顶不住，还是 MySQL 顶不住？

#### 回答框架

我会按链路分层看：

1. **HTTP 入口层**：Tomcat 拒绝连接、RT 抬高、线程池打满
2. **Redis 层**：Lua 执行耗时升高、连接池等待、命令超时
3. **MQ 层**：积压上升、消费者 ACK 变慢、重试增多
4. **MySQL 层**：连接池耗尽、SQL RT 抬高、慢 SQL / 锁等待增加

#### 对应你项目的具体判断法

- 如果前台 `createOrder()` RT 抬高，但 MQ 堆积不明显，更像是**Redis / Tomcat 入口瓶颈**
- 如果前台成功较多，但异步订单长时间不出现，更像是**MQ 消费或 MySQL 落库瓶颈**
- 如果监听器大量 NACK 重试，并出现 JDBC 异常，更像是**MySQL 顶不住**
- 如果 RabbitMQ 队列积压明显、消费者线程利用率不足，要看是**prefetch 太小**还是**消费端处理慢**

### 埋点 10：你调了哪些参数，为什么调这些？

从 `trade-service/src/main/resources/application.yaml` 可以确认你做过这些调优：

#### Tomcat

- `max-threads: 400`
- `min-spare: 50`
- `max-connections: 8192`
- `accept-count: 1000`
- `connection-timeout: 2000`
- `keep-alive-timeout: 20s`
- `max-keep-alive-requests: 1000`

#### RabbitMQ

- `prefetch: 50`
- `concurrency: 16`
- `max-concurrency: 64`
- 手动 ACK
- 关闭 Spring AMQP 自动 retry，自己控制 NACK / requeue

#### Redis Lettuce

- `max-active: 64`
- `max-idle: 32`
- `min-idle: 16`
- `max-wait: 3000ms`

#### 推荐回答

- Tomcat 参数用于提高接入层承载，避免请求大量死在建连或排队阶段
- MQ `prefetch` 增大是为了减少频繁拉取带来的网络往返，提升吞吐
- 并发消费者数量提高是为了增加异步订单消费能力
- 手动 ACK 是为了把成功/失败边界掌握在业务代码里，避免自动确认导致消息误丢
- Redis 连接池扩容是为了避免高并发脚本执行时连接等待时间过长 

#### 追问：prefetch 为什么不是越大越好？

因为太大可能导致：

- 单消费者囤积太多未 ACK 消息
- 处理慢时造成消息分配不均
- 单个消费者崩溃时重投批量更大
- 内存占用增加

所以 `prefetch` 不是越大越好，而是需要结合**消息体大小、单条处理耗时、消费者数量**做平衡。
