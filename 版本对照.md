你的判断是对的，而且这是库存一致性设计里非常关键的一点：**Redis 的 epoch/seq 不能反向污染 MySQL 的事实状态**。否则对账任务就失去了判断“Redis 是否发生过回退/世代切换”的能力。

我按“大厂视角 + 你当前项目结构”给一个更稳的方案。

## 1. 核心结论

### 结论一：`ItemServiceImpl` 不应该直接读取 Redis 的 epoch 写入 MySQL

你现在质疑的点是对的。

如果在 `ItemServiceImpl.deductStock()` 里这样做：

```java
Long epoch = stringRedisTemplate.opsForValue().increment(STOCK_EPOCH_KEY);
Long seq = stringRedisTemplate.opsForValue().increment(STOCK_SEQ_KEY);
deductLog.setEpoch(epoch);
deductLog.setSeq(seq);
```

或者读取 Redis 当前 epoch 后写入 MySQL，那么会有一个致命问题：

> Redis 发生主从切换、回退、重建、恢复之后，MySQL 里的版本也被 Redis 带着走，最终对账任务无法判断 Redis 到底有没有“回到过去”。

正确方向应该是：

> Redis 维护 Redis 自己的版本视图；MySQL/item 服务维护 MySQL 自己的版本视图。两者只能对比，不能互相覆盖版本来源。

---

### 结论二：`StockDeductLog` 不适合承担“全局版本状态机”

你把 `StockDeductLog` 当库存扣减幂等状态机是合理的。

它最适合做这些事情：

1. `orderId` 唯一索引，防止重复扣减；
2. 记录本次扣减了多少；
3. 记录本次扣减状态：已扣减、已回滚、异常；
4. 作为审计流水；
5. 为人工排查、补偿任务提供依据。

但它不适合承担：

1. 当前 MySQL 库存版本；
2. 当前 MySQL epoch；
3. 当前对账退避状态；
4. Redis/Mysql 双边版本比较状态。

因为它是**事件流水表**，不是**状态表**。

大厂里通常会区分：

| 类型 | 表 | 特点 |
|---|---|---|
| 业务事实表 | `item` | 当前真实库存 |
| 事件流水表 | `stock_deduct_log` | 每笔订单扣减记录、幂等依据 |
| 版本状态表 | `item_stock_version` 或 `stock_version_state` | 当前 MySQL 侧库存版本 |
| 对账状态表 | `stock_reconcile_state` | 差异检测、退避、修复状态 |

---

### 结论三：不建议把复杂版本字段直接塞进 `Item`

可以在 `Item` 上维护每个商品的版本，比如：

```text
item.stock_epoch
item.stock_seq
item.stock_version
```

这个方案不是不行，但对你当前项目来说，我不推荐作为第一选择。

原因：

1. `item` 表是商品主表，已经承担了业务属性；
2. 每次库存扣减都要同时更新 `item.stock` 和 `item.version`；
3. 批量扣减多个商品时，版本更新 SQL 会变复杂；
4. 版本字段和商品业务字段混在一起，长期会污染模型；
5. 后续如果要做 Redis/MySQL/Canal/XXL-JOB 对账，会越来越难维护。

更合理的是：**单独建 item 服务侧库存版本状态表**。

## 3. Redis 侧版本应该怎么定义？

Redis 侧可以继续维护自己的版本：

```text
item:stock:epoch:{stock}
item:stock:seq:{stock}
item:stock:ver:{stock}:{itemId}
item:stock:{stock}:{itemId}
```

但 Redis 的版本只代表：

> Redis 这份缓存库存当前处于什么世代、什么序列。

不能代表 MySQL。

Lua 预扣库存时，可以做：

1. 检查库存是否充足；
2. 扣减 Redis 库存；
3. `redis_seq++`；
4. 每个被扣商品写入 `item:stock:ver:{stock}:{itemId} = redis_epoch|redis_seq`；
5. 写入 Redis outbox。

注意：这里 Redis 写出的 `redis_epoch|redis_seq` 只用于后续对账观察，不能直接传给 MySQL 当作 MySQL 版本。

## 4. MySQL 扣减库存时应该怎么维护版本？

`ItemServiceImpl.deductStock()` 的职责应该是：

1. 插入 `stock_deduct_log`，用 `order_id` 唯一键做幂等；
2. 执行 `item.stock = item.stock - num`；
3. 同事务内更新 `item_stock_version`；
4. 提交后 ACK MQ。

也就是说，MySQL 侧版本应该跟 MySQL 库存更新在同一个事务里。

伪流程：

```text
收到 ORDER_CREATED MQ
    |
    |-- insert stock_deduct_log(order_id, status=PROCESSING)
    |
    |-- update item set stock = stock - num where stock >= num
    |
    |-- update item_stock_version
            set mysql_seq = mysql_seq + 1,
                last_order_id = orderId,
                last_event_type = 'DEDUCT'
            where item_id = ?
    |
    |-- update stock_deduct_log status=SUCCESS
    |
事务提交
    |
ACK MQ
```

这里有一个关键点：

> MySQL epoch 不应该因为普通扣减增长。普通扣减只增长 `mysql_seq`。

`mysql_epoch` 什么时候变？

只有在 MySQL 侧发生“世代切换”时才变，例如：

1. 人工全量重建库存；
2. 业务做库存盘点并重置；
3. MySQL 主从恢复后需要明确标记新世代；
4. 数据修复任务认为当前 MySQL 库存事实进入了新版本周期。

普通扣减和退还库存都只增加 `mysql_seq`。

## 7. 对账任务应该怎么做？

`StockReconciliationJobHandler` 应该做的是：

```text
读取 MySQL item + item_stock_version
读取 Redis stock + redis version
比较两边状态
根据差异类型决定：观察 / 退避 / 立即修复
```

### 差异类型一：Redis 库存小于 MySQL

```text
redis_stock < mysql_stock
```

这通常说明：

1. Redis 已经预扣；
2. MQ 还没消费到 item 服务；
3. MySQL 还没扣；
4. 或者 MQ 堆积。

这类情况**不能立即用 MySQL 覆盖 Redis**。

否则会把正在飞行中的预扣库存加回去，造成超卖。

正确处理：

```text
记录对账状态
指数退避
观察多轮
超过安全窗口后再修复
```

这也是你之前对“飞行消息”的担心，是正确的。

---

### 差异类型二：Redis 库存大于 MySQL

```text
redis_stock > mysql_stock
```

这个非常危险。

因为 Redis 前置预扣系统里，正常情况下：

```text
Redis 库存 <= MySQL 库存
```

如果 Redis 反而比 MySQL 大，可能说明：

1. Redis 主从切换丢了扣减；
2. Redis 被重建但用了旧数据；
3. 手动误操作；
4. 缓存回填覆盖了较新的预扣状态。

这种情况应该：

```text
立即以 MySQL 覆盖 Redis
```

因为继续放任 Redis 大于 MySQL，会直接超卖。

---

### 差异类型三：Redis 版本缺失

```text
redis_version == null
```

如果 Redis stock 存在但 version 缺失，说明缓存状态不完整。

处理方式：

1. 如果 `redis_stock > mysql_stock`：立即修复；
2. 如果 `redis_stock <= mysql_stock`：进入退避观察；
3. 多轮稳定后以 MySQL 回填。

---

### 差异类型四：Redis epoch 变化

这个最关键。

假设：

```text
MySQL: mysql_epoch = 1, mysql_seq = 10000
Redis: redis_epoch = 2, redis_seq = 50
```

这不一定是坏事。

Redis epoch 增大，可能说明 Redis 做过重建/故障切换。

问题是：

> Redis epoch 增大以后，Redis stock 是否安全？

如果 Redis stock 大于 MySQL stock，则危险，立即修复。

如果 Redis stock 小于 MySQL stock，可能只是预扣领先，可以退避观察。

所以大厂不会单纯因为 `redis_epoch != mysql_epoch` 就覆盖，而是结合库存大小、版本顺序、飞行消息窗口综合判断。

## 8. 推荐的对账状态表

建议新增：

```text
stock_reconcile_state
```

用于维护对账退避状态，而不是放 Redis 里。

你现在 `StockReconciliationJobHandler` 把 `retryCount`、`nextRetryAt` 放 Redis hash 里：

```java
stringRedisTemplate.opsForHash().put(reconcileKey, "retryCount", ...);
```

从严格一致性角度看，这也不太理想。

因为对账状态本身是“纠偏控制平面”，应该放 MySQL，不能依赖被对账对象 Redis。

推荐表：

```sql
CREATE TABLE stock_reconcile_state (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    item_id BIGINT NOT NULL,
    mysql_stock INT NOT NULL,
    redis_stock INT DEFAULT NULL,
    mysql_epoch BIGINT NOT NULL,
    mysql_seq BIGINT NOT NULL,
    redis_epoch BIGINT DEFAULT NULL,
    redis_seq BIGINT DEFAULT NULL,
    diff_type VARCHAR(64) NOT NULL,
    retry_count INT NOT NULL DEFAULT 0,
    next_retry_at DATETIME DEFAULT NULL,
    status TINYINT NOT NULL DEFAULT 0 COMMENT '0-观察中,1-已修复,2-忽略,3-人工介入',
    last_check_time DATETIME NOT NULL,
    create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uk_item_id (item_id)
);
```

这样对账任务本身不依赖 Redis 记录状态。
