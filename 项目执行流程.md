# Redis / Lua / MySQL / XXL-JOB 最终落地清单

## 1. 总体目标

本项目采用“Redis 快路径 + Redis Outbox 临时缓冲 + MySQL Outbox 最终兜底 + RabbitMQ 异步分发 + XXL-JOB 补偿与对账”的最终一致性方案。

核心目标：

- Redis 保证高并发下单的库存扣减性能
- Lua 脚本保证“扣库存 + 写 Outbox”原子性
- RabbitMQ 负责异步事件分发
- Redis Outbox 负责极端掉电/发送失败兜底
- MySQL Outbox 与库存流水负责最终可靠性与幂等
- Epoch + Sequence 负责 Redis 故障切换后的版本判断与对账收敛

---

## 2. Redis Key 设计

### 2.1 库存 key

建议保留 hash tag，确保 Lua 相关 key 落在同一 slot：

- `item:stock:{stock}:itemId`

用途：
- 存储商品实时库存
- 支持 Lua 原子扣减

---

### 2.2 库存版本 key

新增版本相关 key：

- `item:stock:epoch:{stock}`
- `item:stock:seq:{stock}`
- `item:stock:ver:{stock}`

含义：

- `epoch`：Redis 故障世代号
- `seq`：当前世代内单调递增序号
- `ver`：逻辑版本号，建议格式 `epoch|seq`

---

### 2.3 Redis Outbox key

沿用分桶设计：

- `trade:local_msg_outbox:{stock}:{bucket}`

用途：
- 保存订单创建消息
- 作为 MQ 发送失败时的临时缓冲区

建议数据结构：
- `HASH`
- field = `orderId`
- value = `msgJson`

---

### 2.4 对账标记 key

可选新增：

- `item:stock:reconcile:{stock}:{itemId}`
- 或 `trade:reconcile:{stock}:{orderId}`

用途：
- 标记该业务是否处于待对账 / 待恢复状态
- 避免重复补偿与重复覆盖

---

## 3. Lua 脚本最终职责

Lua 脚本负责一次性完成以下动作：

1. 校验库存是否足够
2. 原子扣减库存
3. 生成 `seq`
4. 读取 `epoch`
5. 生成 `version = epoch|seq`
6. 写入 Redis Outbox
7. 写入库存版本标记

---

## 4. Lua 输入输出约定

### 4.1 KEYS

- `KEYS[1..n]`：多个商品库存 key
- `KEYS[n+1]`：Redis Outbox key
- `KEYS[n+2]`：epoch key
- `KEYS[n+3]`：seq key

### 4.2 ARGV

- `ARGV[1..n]`：每个商品扣减数量
- `ARGV[n+1]`：orderId
- `ARGV[n+2]`：消息 JSON

### 4.3 返回值建议

建议返回结构化结果，便于 Java 层判定：

- `0`：成功
- `-1`：库存不足
- `-2`：epoch 不匹配
- `-3`：参数错误

也可以返回失败商品 id，便于日志排查。

## 5. MySQL 表设计

## 5.1 `local_event_outbox`（trade 服务）

对应实体：`trade-service/src/main/java/com/hmall/trade/domain/po/LocalEventOutbox.java`

建议字段：

- `id`
- `orderId`：业务主键，例如 `orderId`
- `event_type`：事件类型，如 `ORDER_CREATED`、`RESTORE_ITEM_STOCK`
- `payload`：事件内容 JSON
- `epoch`
- `seq`
- `version`
- `source`：`REDIS` / `MYSQL`

- `retry_count`
- `next_retry_time`
- `create_time`
- `update_time`

### 作用

- 作为最终可靠的本地消息表
- 记录订单创建、库存恢复等关键业务事件
- 作为 Redis Outbox 的最终兜底

## 5.2 `stock_deduct_log`（item 服务）

对应实体：`item-service/src/main/java/com/hmall/item/domain/po/StockDeductLog.java`

建议字段：

- `id`
- `order_id`
- `item_id`
- `deduct_num`
- `epoch`
- `seq`
- `version`
- `status`
- `create_time`

### 作用

- 作为库存扣减幂等依据
- 作为对账和恢复库存的事实流水

## 6. trade 服务落地职责

### 6.1 `OrderServiceImpl.createOrder`

职责：

1. 生成 `orderId`
2. 调用 Redis Lua 脚本
3. 在 Lua 内完成扣库存与写 Redis Outbox
4. 尽最大努力同步发 RabbitMQ
5. 成功则删除 Redis Outbox
6. 失败交给 XXL-JOB 兜底

---

### 6.2 `MqCompensationXxlJob`

职责：

1. 扫描 Redis 分桶 Outbox
2. 取出待发送消息
3. 重新投递 RabbitMQ
4. 成功删除 Redis Outbox
5. 失败保留，等待下次任务重试

---

### 6.3 `LocalEventOutbox`

职责：

- 记录 trade 服务内的最终业务事件
- 为 item/cart/delay 等下游消息提供最终兜底

---

## 7. item 服务落地职责

### 7.1 `ItemServiceImpl.deductStock`

职责：

1. 插入 `stock_deduct_log`
2. 依赖唯一索引实现强幂等
3. 更新数据库真实库存
4. 为 Redis / MQ 对账提供可追溯流水

### 7.2 `ItemServiceImpl.increaseStock`

职责：

1. 先把 `stock_deduct_log` 状态从 1 改为 2
2. 通过状态机判断是否已恢复过库存
3. 批量恢复数据库库存
4. 再通过补偿机制恢复 Redis 库存

---

## 8. XXL-JOB 最终拆分建议

### 8.1 Redis Outbox 补偿任务

对应现有：`MqCompensationXxlJob`

职责：
- 扫 Redis Outbox
- 重发 MQ
- 成功删除 Redis Outbox

---

### 8.2 MySQL Outbox 补偿任务

职责：
- 扫描 `local_event_outbox`
- 发送订单创建、库存恢复等事件到 MQ
- 成功后更新状态
- 失败后记录重试次数和下次重试时间

---

### 8.3 Redis / MySQL 对账恢复任务

职责：
- 比较 Redis epoch / seq 与 MySQL 侧流水
- 判断 Redis 是否经历过故障切换
- 如果仍存在 MQ 飞行消息，不立即强覆盖 Redis
- 采用指数退避延迟重试
- 收敛后才用 MySQL 事实源回填 Redis

---

### 8.4 Epoch 迁移任务

职责：
- 检测 Redis 故障切换或关键版本 key 丢失
- epoch 自增
- 进入新世代写入
- 避免旧世代数据与新世代数据混淆

---

## 9. 执行顺序建议

### 第一阶段
- 补齐 Redis 版本 key
- 升级 Lua 脚本
- 给 Redis Outbox 写入版本信息

### 第二阶段
- 改造 `LocalEventOutbox`
- 改造 `StockDeductLog`
- 让 MySQL 记录具备版本号与来源标识

### 第三阶段
- 强化 `MqCompensationXxlJob`
- 拆分 MySQL Outbox 补偿任务

### 第四阶段
- 实现对账恢复任务
- 实现 epoch 切换与指数退避

## 10. 面试表述建议

可以这样描述：

> 我们的方案不是单纯依赖 Redis 或 MQ，而是采用 Redis 快路径、Redis Outbox 临时缓冲、MySQL Outbox 最终兜底、RabbitMQ 异步分发的最终一致性架构。Lua 脚本只保证原子性，不保证持久性，因此 Redis Outbox 仅作为高性能短链路缓冲；MySQL 的本地消息表和库存流水才是最终事实来源。为了处理 Redis 宕机和主从切换后的误判问题，引入 epoch + sequence 的逻辑版本号，恢复时先比 epoch，再比 sequence，最后结合 MQ 飞行消息和 MySQL 流水做延迟对账与指数退避收敛。

## 12. 对应的 MySQL 建表字段补充建议

### 12.1 `local_event_outbox` 表建议新增字段

```sql
ALTER TABLE local_event_outbox
  ADD COLUMN order_id BIGINT NULL COMMENT '业务主键，例如 orderId',
  ADD COLUMN epoch BIGINT NOT NULL DEFAULT 1 COMMENT 'Redis 故障世代号',
  ADD COLUMN seq BIGINT NOT NULL DEFAULT 0 COMMENT '当前世代内单调递增序号',
  ADD COLUMN version VARCHAR(64) NOT NULL DEFAULT '' COMMENT '逻辑版本号，建议格式 epoch|seq',
  ADD COLUMN source VARCHAR(16) NOT NULL DEFAULT 'REDIS' COMMENT '消息来源：REDIS/MYSQL',
  ADD COLUMN retry_count INT NOT NULL DEFAULT 0 COMMENT '重试次数',
  ADD COLUMN next_retry_time DATETIME NULL COMMENT '下次重试时间';
```

建议再补充索引：

```sql
CREATE INDEX idx_local_event_outbox_status_retry
ON local_event_outbox(status, next_retry_time);

CREATE INDEX idx_local_event_outbox_order_id
ON local_event_outbox(order_id);

CREATE INDEX idx_local_event_outbox_version
ON local_event_outbox(epoch, seq);
```

### 12.2 `stock_deduct_log` 表建议新增字段

```sql
ALTER TABLE stock_deduct_log
  ADD COLUMN item_id BIGINT NOT NULL COMMENT '商品ID',
  ADD COLUMN deduct_num INT NOT NULL COMMENT '扣减数量',
  ADD COLUMN epoch BIGINT NOT NULL DEFAULT 1 COMMENT 'Redis 故障世代号',
  ADD COLUMN seq BIGINT NOT NULL DEFAULT 0 COMMENT '当前世代内单调递增序号',
  ADD COLUMN version VARCHAR(64) NOT NULL DEFAULT '' COMMENT '逻辑版本号，建议格式 epoch|seq';
```

建议再补充索引：

```sql
CREATE UNIQUE INDEX uk_stock_deduct_log_order_item
ON stock_deduct_log(order_id, item_id);

CREATE INDEX idx_stock_deduct_log_version
ON stock_deduct_log(epoch, seq);
```

### 12.3 字段含义补充说明

- `order_id`：用于识别业务主键，方便幂等和对账。
- `epoch`：Redis 故障或主从切换后的“世代号”。
- `seq`：同一世代内的单调递增序号。
- `version`：对外展示的完整版本号，建议拼成 `epoch|seq`。
- `source`：标识这条 outbox 来源于 Redis 还是 MySQL。
- `retry_count` / `next_retry_time`：用于补偿任务的退避重试。

## 13. 结论

最终推荐的落地方案是：

- **Redis**：高性能扣减 + 临时 outbox + 版本标记
- **MySQL**：最终消息表 + 库存流水表
- **RabbitMQ**：异步事件传输
- **XXL-JOB**：补偿、对账、收敛、恢复

这套方案既适合当前项目代码结构，也适合面试表达。
